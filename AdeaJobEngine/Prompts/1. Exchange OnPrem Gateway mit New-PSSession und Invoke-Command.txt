O = Ziel

Baue die Exchange-OnPrem-Verbindungslogik so um, dass ExchangeOnPremGateway.psm1 autonom sicherstellt, dass eine gültige Exchange-OnPrem-Remote-PowerShell-Session existiert.

Aktuell prüft ExchangeOnPremGateway.psm1 nur über Assert-OnPremCmdlet, ob lokale Exchange-Cmdlets wie Get-Mailbox, Set-Mailbox oder Add-MailboxPermission bereits vorhanden sind. Wenn nicht, wird ein Fehler geworfen.

Das darf so nicht bleiben.

Wenn ein Exchange-OnPrem-Befehl ausgeführt wird, muss das Gateway selbst:

eine Remote-PowerShell-Session prüfen,
bei Bedarf eine neue Session aufbauen,
die Session wiederverwenden,
defekte oder abgelaufene Sessions neu aufbauen,
Exchange-Befehle über Invoke-Command ausführen.

────────────────────────────────────────

K = Hintergrund

Die alten Legacy-Skripte verwendeten Import-PSSession. Das funktioniert, ist aber langsam.

Gemessene Werte:

New-PSSession plus Import-PSSession:
ca. 10.5 Sekunden

New-PSSession ohne Import-PSSession:
ca. 2.2 Sekunden

Deshalb soll Import-PSSession nicht mehr der Standardweg sein.

Die neue Standardvariante ist:

New-PSSession erstellen
keine lokalen Proxy-Cmdlets importieren
Exchange-Befehle über Invoke-Command -Session ausführen

Wichtig:

New-PSSession allein stellt lokale Cmdlets wie Get-Mailbox nicht lokal bereit. Deshalb dürfen Gateway-Funktionen im InvokeCommand-Modus nicht mehr darauf bauen, dass Get-Mailbox lokal verfügbar ist.

────────────────────────────────────────

A = Architekturvorgaben

Die technische Zielkette ist:

Service
  → ExchangeOnPremGateway
    → Ensure Exchange OnPrem Session
    → Invoke-Command gegen Remote-PSSession
    → Exchange Cmdlet im Remote-Session-Kontext ausführen

Nicht erlaubt:

direkte Exchange-Cmdlets in Handlern
direkte Exchange-Cmdlets in Services
produktive Werte im Code hardcoden
OnPrem-Logik mit Exchange-Online-Logik vermischen
ExchangeOnlineGateway in diesem Schritt fachlich umbauen
fachliche Hybrid-Routing-Logik ändern
UseCases ändern, ausser es ist zwingend nötig
Legacy-Skripte verändern

Erforderliche Funktionen

Implementiere mindestens folgende Funktionen:

Get-ExchangeOnPremRemotePowerShellConfig

Aufgabe:

RemotePowerShell-Konfiguration aus Context.Config oder übergebenem Config-Objekt lesen.
Pflichtfelder validieren.
Kontrollierte Fehler erzeugen, wenn Enabled = false oder Pflichtwerte fehlen.

────────────────────────────────────────

New-ExchangeOnPremCredential

Aufgabe:

User und SecretPath aus Config verwenden.
SecretPath lesen.
ConvertTo-SecureString verwenden.
PSCredential erzeugen.
Kein Passwort hardcoden.
Klare Fehlermeldung, wenn SecretPath nicht existiert oder nicht lesbar ist.

────────────────────────────────────────

Connect-ExchangeOnPremSession

Aufgabe:

New-PSSession mit ConfigurationName Microsoft.Exchange aufbauen.
ConnectionUri, Authentication und Credential aus Config verwenden.
PSSessionOption anhand SessionIdleTimeoutMinutes erstellen, falls sinnvoll.
Session-Metadaten speichern:
    CreatedAt
    LastUsedAt
    ConnectionUri
    User
    ExecutionMode

Wichtig:

Default ist UseImportPSSession = false.
Default ist ExecutionMode = InvokeCommand.
Import-PSSession nur ausführen, wenn UseImportPSSession = true konfiguriert ist.

────────────────────────────────────────

Test-ExchangeOnPremSession

Aufgabe:

Prüfen, ob Sessionobjekt vorhanden ist.
Prüfen, ob Session.State Opened ist.
Prüfen, ob MaxSessionAgeMinutes noch nicht überschritten ist.
Optional leichten Remote-Test ausführen, wenn nötig.

────────────────────────────────────────

Get-ExchangeOnPremSession

Aufgabe:

Bestehende gültige Session zurückgeben.
Wenn keine gültige Session vorhanden ist, Connect-ExchangeOnPremSession aufrufen.
Wenn ReuseSession = false, immer neue Session aufbauen.

────────────────────────────────────────

Invoke-ExchangeOnPremCommand

Aufgabe:

Sicherstellen, dass eine gültige Session vorhanden ist.
ScriptBlock remote über Invoke-Command ausführen.
Argumente sauber per ArgumentList übergeben.
LastUsedAt aktualisieren.
Bei Sessionfehler:
    wenn ReconnectOnFailure = true:
        Session resetten,
        einmal neu verbinden,
        Befehl einmal erneut versuchen.
    sonst:
        kontrollierten Fehler werfen.

────────────────────────────────────────

Reset-ExchangeOnPremSession

Aufgabe:

Vorhandene PSSession entfernen.
Lokale Cache-Variablen leeren.
Importiertes Modul nur entfernen, falls UseImportPSSession verwendet wurde.

────────────────────────────────────────

Disconnect-ExchangeOnPremSession

Aufgabe:

PSSession sauber entfernen.
Cache leeren.
Keine Fehler werfen, wenn Session bereits weg ist.

────────────────────────────────────────

Bestehende Gateway-Funktionen anpassen

Prüfe vorhandene Funktionen in ExchangeOnPremGateway.psm1, zum Beispiel:

Get-OnPremMailboxSafe
Set-OnPremMailboxSafe
Add-OnPremMailboxPermissionSafe
Remove-OnPremMailboxPermissionSafe
Add-OnPremRecipientPermissionSafe
Remove-OnPremRecipientPermissionSafe
Get-OnPremRecipientSafe
Set-OnPremRecipientSafe

Diese Funktionen dürfen im Standardmodus nicht mehr voraussetzen, dass lokale Exchange-Cmdlets importiert wurden.

Bei ExecutionMode = InvokeCommand müssen sie intern Invoke-ExchangeOnPremCommand verwenden.

Beispielmuster:

Funktion im Gateway
  → Invoke-ExchangeOnPremCommand
    → Remote ScriptBlock
      → Get-Mailbox / Set-Mailbox / Add-MailboxPermission im Remote-Kontext

Assert-OnPremCmdlet darf bestehen bleiben, aber nur noch:

für optionalen ImportPSSession-Modus,
für Diagnostik,
nicht als einzige Voraussetzung für OnPrem-Exchange-Befehle.

Pester-Modus und Mockbarkeit

Folgende Cmdlets müssen mockbar sein:

New-PSSession

Beispiel für Wrapper-Namen:

Import-ExchangeOnPremPSSessionInternal

Diese Wrapper dürfen keine Fachlogik enthalten. Sie dienen nur der Testbarkeit.

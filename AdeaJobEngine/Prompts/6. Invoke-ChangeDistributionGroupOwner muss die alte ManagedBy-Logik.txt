────────────────────────────────────────

OwnerIdentity / ManagedByMembers parsing

Die alte Logik verarbeitet Einträge mit Suffix:

[ADD]
[DEL]

Beispiele:

user1[ADD]
user2[DEL]

Die neue Logik soll diese Semantik übernehmen.

Parsing-Regel:

Wenn OwnerIdentity oder ManagedByMember mit [ADD] endet:
    Operation = Add
    MemberIdentity = Wert ohne [ADD]

Wenn OwnerIdentity oder ManagedByMember mit [DEL] endet:
    Operation = Remove
    MemberIdentity = Wert ohne [DEL]

Wenn kein Suffix vorhanden ist:
    kontrollierter Fehler oder, falls bestehende Fachlogik das erwartet, Default = Add

Empfehlung:

Kein stiller Default, ausser bestehende Tests/Legacy-Daten verlangen es.
Besser kontrollierten Fehler mit klarer Meldung erzeugen:
OwnerIdentity must end with [ADD] or [DEL].

Mehrfachwerte:

Falls OwnerIdentity mehrere Werte enthält, prüfe bestehende CSV-Konvention.

Unterstütze nur vorhandene Konventionen, zum Beispiel:

Semikolon-getrennt
Komma-getrennt
Pipe ist Feldtrenner und darf nicht als Mehrwerttrenner verwendet werden
Array aus Payload

Keine neue Mehrwertsyntax erfinden, wenn im bestehenden Projekt bereits eine Konvention vorhanden ist.

Exchange OnPrem Mutation

Die fachliche Operation ist:

oder:

Set-DistributionGroup -Identity GroupIdentity -ManagedBy @{Remove=MemberIdentity} -BypassSecurityGroupManagerCheck

DomainController:

Die alte Legacy-Logik verwendete:

Wenn das neue ExchangeOnPremGateway bereits eine DomainController-Konfiguration unterstützt, verwende sie.

Wenn nicht, nicht hart codieren.

Keine produktiven DC-Werte hardcoden.

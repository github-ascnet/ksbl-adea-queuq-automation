────────────────────────────────────────

O = Ziel

Ersetze die aktuelle Platzhalterlogik in Get-NextAvailableAccountName durch eine echte, prefixbasierte sAMAccountName-Generierung.

Aktueller Zustand:

function Get-NextAvailableAccountName {
    [CmdletBinding()]
    param([Parameter(Mandatory = $true)][string]$BaseName)

    # TODO: Migrate legacy logic here
    # Placeholder strategy: return base name as-is.
    ConvertTo-SafeAccountName -InputValue $BaseName
}

Dieser Placeholder ist fachlich falsch, weil er keinen freien Accountnamen im On-Prem Active Directory ermittelt.

Ziel ist eine deterministische, testbare und AD-gestützte Funktion:

Get-NextAvailableSamAccountName

Diese Funktion ermittelt für die erlaubten Prefixe gmb, us und ex den nächsten freien sAMAccountName anhand der bestehenden Namen im On-Prem Active Directory.

────────────────────────────────────────

K = Verbindliche Master-Regel

Für die Generierung von sAMAccountName ist ausschliesslich das On-Prem Active Directory der Master.

Die Prüfung auf bestehende Namen erfolgt nur gegen das lokale On-Prem AD.

Nicht prüfen gegen:

Exchange Online
Entra ID
Microsoft 365
Exchange OnPrem Recipient Cmdlets
Exchange Online Recipient Cmdlets
SQL-Datenbank als führende Quelle
CSV-Dateien
Cache-Dateien

Die SQL-Datenbank kann nachgelagert Daten aus dem AD erhalten, ist aber nicht die führende Quelle für die Eindeutigkeit von sAMAccountName.

Die Exchange- oder M365-Welt darf bei dieser Funktion keine Rolle spielen.

Zielregel:

Get-NextAvailableSamAccountName
  → liest bestehende sAMAccountName-Werte aus dem On-Prem AD
  → ermittelt den nächsten freien Wert im definierten Nummernkreis
  → gibt den neuen Namen zurück

────────────────────────────────────────

K = Fachliche Prefix-Vorgabe

Es gibt genau drei erlaubte Prefixe.

Prefix	Nummernkreis	Startwert	Format	Beispiel
gmb	4-stellig	1	gmb0001	gmb0001
us	5-stellig	10000	us10000	us10000
ex	5-stellig	100	ex00100	ex00100

Logik pro Prefix:

gmb
  StartValue = 1
  Width = 4
  MaxValue = 9999

us
  StartValue = 10000
  Width = 5
  MaxValue = 99999

ex
  StartValue = 100
  Width = 5
  MaxValue = 99999

Der Prefix ist case-insensitive zu behandeln, das Ergebnis soll aber immer lower-case zurückgegeben werden.

Beispiele:

Get-NextAvailableSamAccountName -Prefix gmb
  → sucht den ersten freien Wert ab gmb0001

Get-NextAvailableSamAccountName -Prefix us
  → sucht den ersten freien Wert ab us10000

Get-NextAvailableSamAccountName -Prefix ex
  → sucht den ersten freien Wert ab ex00100

Get-NextAvailableSamAccountName -Prefix gmb -UseHighestPlusOne
  → nimmt den höchsten bestehenden gmb-Wert und addiert 1

────────────────────────────────────────

AD-Suche sauber kapseln

Die Suche nach bestehenden Accounts muss gegen das On-Prem Active Directory erfolgen.

Wenn ActiveDirectoryGateway.psm1 vorhanden ist, erweitere es um eine kleine, klar benannte Funktion, zum Beispiel:

Get-ExistingSamAccountNamesByPrefix

oder:

Get-AdSamAccountNamesByPrefix

Aufgabe dieser Gateway-Funktion:

On-Prem AD nach sAMAccountName-Werten mit dem Prefix durchsuchen
nur sAMAccountName-Werte zurückgeben
keine Namensgenerierungslogik enthalten
keine Exchange-/M365-/SQL-Logik enthalten

Beispielhafte interne Logik im Gateway:

LDAP-Filter auf User-Accounts mit sAMAccountName prefix*
Properties: sAMAccountName
Rückgabe: Liste von Strings

Der LDAP-Filter soll sinngemäss auf User-Accounts zielen:

sAMAccountType = 805306368
sAMAccountName beginnt mit Prefix

Wichtig:

Get-ADUser oder DirectorySearcher darf nur in der Gateway-Schicht verwendet werden.
Die Namenslogik selbst soll testbar bleiben.
AD-Zugriffe müssen in Pester gemockt werden können.

────────────────────────────────────────

Bestehende Namen auswerten

Die Logik darf nur bestehende Namen berücksichtigen, die exakt dem Muster entsprechen:

Prefix plus Ziffern

Gültige Beispiele:

gmb0001
gmb1234
us10000
us54321
ex00100
ex99999

Ungültige Beispiele, die ignoriert werden müssen:

gmb-test
gmbABC
gmb0001_old
usadmin
ex00100_old
user00100
adm00013

Regex-Logik sinngemäss:

^prefix(\d+)$

Die extrahierten Nummern werden als Integer interpretiert.

Nur Nummern zwischen StartValue und MaxValue berücksichtigen.

Doppelte Werte entfernen.

Aufsteigend sortieren.

────────────────────────────────────────

Standardlogik: erste freie Lücke suchen

Ohne UseHighestPlusOne gilt:

Starte bei StartValue.
Gehe durch die sortierten bestehenden Nummern.
Wenn bestehende Nummer gleich nextNumber ist:
  nextNumber erhöhen.
Wenn bestehende Nummer grösser als nextNumber ist:
  Lücke gefunden, abbrechen.
Ergebnis ist Prefix plus gepaddete nextNumber.

Beispiele:

Prefix gmb
vorhanden: keine
Ergebnis: gmb0001

Prefix gmb
vorhanden: gmb0001, gmb0002
Ergebnis: gmb0003

Prefix gmb
vorhanden: gmb0001, gmb0002, gmb0004
Ergebnis: gmb0003

Prefix us
vorhanden: keine
Ergebnis: us10000

Prefix ex
vorhanden: ex00100, ex00101
Ergebnis: ex00102

────────────────────────────────────────

UseHighestPlusOne-Logik

Wenn UseHighestPlusOne gesetzt ist:

Wenn bestehende gültige Nummern vorhanden sind:
  höchste Nummer nehmen und plus 1 rechnen.
Wenn keine bestehenden gültigen Nummern vorhanden sind:
  StartValue verwenden.

Beispiele:

Prefix gmb
vorhanden: gmb0001, gmb0002, gmb0004
Ergebnis: gmb0005

Prefix us
vorhanden: us10000, us10003
Ergebnis: us10004

Prefix ex
vorhanden: keine
Ergebnis: ex00100

────────────────────────────────────────

Nummernkreis erschöpft

Wenn die berechnete nextNumber grösser als MaxValue ist, muss ein kontrollierter Fehler geworfen werden.

Fehlermeldung sinngemäss:

Der Nummernkreis für Prefix 'gmb' ist ausgeschöpft. Maximalwert: 9999.

Diese Prüfung gilt sowohl für die Standardlogik als auch für UseHighestPlusOne.

────────────────────────────────────────

Rückwärtskompatibilität für Get-NextAvailableAccountName

Prüfe, ob Get-NextAvailableAccountName weiterhin benötigt wird.

Falls ja, implementiere diese Funktion als Wrapper.

Der Wrapper soll:

BaseName trimmen
Prefix aus BaseName ableiten
bei gmb... Prefix gmb verwenden
bei us... Prefix us verwenden
bei ex... Prefix ex verwenden
Get-NextAvailableSamAccountName -Prefix Prefix aufrufen
bei unbekanntem Prefix einen kontrollierten Fehler werfen

Beispiele:

Get-NextAvailableAccountName -BaseName "gmb"
  → ruft Get-NextAvailableSamAccountName -Prefix gmb

Get-NextAvailableAccountName -BaseName "gmbTest"
  → ruft Get-NextAvailableSamAccountName -Prefix gmb

Get-NextAvailableAccountName -BaseName "us"
  → ruft Get-NextAvailableSamAccountName -Prefix us

Get-NextAvailableAccountName -BaseName "ex00100"
  → ruft Get-NextAvailableSamAccountName -Prefix ex

Get-NextAvailableAccountName -BaseName "admin"
  → kontrollierter Fehler

Wichtig:

Der Wrapper darf nicht mehr ConvertTo-SafeAccountName als Ersatzstrategie verwenden.

Wenn alle Aufrufer sauber auf Get-NextAvailableSamAccountName umgestellt werden können, darf Get-NextAvailableAccountName als deprecated Wrapper bestehen bleiben, damit alte Pfade nicht brechen.

Wichtigste Zielregel

Die sAMAccountName-Generierung muss deterministisch, prefixbasiert, testbar und ausschliesslich On-Prem-AD-gestützt sein.

Master für sAMAccountName ist immer das On-Prem Active Directory.

gmb beginnt bei gmb0001.
us beginnt bei us10000.
ex beginnt bei ex00100.

Ohne UseHighestPlusOne wird die erste freie Lücke gesucht.

Mit UseHighestPlusOne wird der höchste bestehende Wert plus 1 verwendet.

Exchange Online, Entra ID, Microsoft 365 und SQL dürfen für diese Entscheidung nicht als führende Quelle verwendet werden.

O = Ziel

Erweitere die bestehende ASP.NET Core MVC Applikation so, dass nach erfolgreichem Erzeugen eines IAM-Queue-Files automatisch die PowerShell-AutomationEngine über Hangfire angesteuert wird.

Der Ablauf soll sein:

1. Benutzer führt eine bestehende HttpPost-Action aus.
2. Controller baut wie bisher IamQueueFileContent.
3. Controller ruft IamQueueService auf und schreibt die Queue-Datei.
4. Direkt nach erfolgreichem Queue-File-Schreiben wird ein Automation Job erzeugt.
5. Hangfire startet im Hintergrund Invoke-JobProcessor.ps1.
6. Invoke-JobProcessor.ps1 wird mit Queue, CorrelationId, OutputJson und möglichst gezieltem QueueFileName gestartet.
7. stdout wird als JSON geparst.
8. stderr wird separat geloggt.
9. Jobstatus und Events werden persistiert.
10. Ein JobController stellt Status und Events per REST bereit.
11. Optional wird SignalR vorbereitet, damit der Client Statusänderungen live erhalten kann.

Wichtig:

Hangfire ersetzt nicht die PowerShell-JobEngine.

Hangfire ist nur der technische Background Runner.

Die fachliche Wahrheit bleibt:

AdeaJobEngine Invoke-JobProcessor.ps1
AdeaJobEngine JobEngine
Queue-Dateien
.meta.json
OutputJson

K = Bestehender PowerShell-Stand

Die separate PowerShell-Automation AdeaJobEngine ist bereits vorbereitet.

Invoke-JobProcessor.ps1 unterstützt:

Queue
OutputJson
CorrelationId

Beispiel:

powershell.exe -NoProfile -ExecutionPolicy Bypass -File Invoke-JobProcessor.ps1 -Queue standard -CorrelationId GUID -OutputJson

Gültige Queue-Werte:

standard
urgent
person-mailbox-longrunning

Ungültig:

longrunning

Die physische Queue-Struktur ist flach:

queues/incoming
queues/processing
queues/done
queues/failed
queues/retry
queues/paused
queues/archive

Es gibt keine physischen Unterordner wie:

standard/incoming

Die fachliche Queue wird über den Parameter Queue bestimmt.

OutputJson liefert am Ende ein maschinenlesbares JSON über stdout.

stdout ist die maschinenlesbare Quelle.

stderr ist nur technischer Diagnosekanal.

CorrelationId wird in der PowerShell-JobEngine als ExternalCorrelationId in die .meta.json übernommen.

Wichtiger Architekturhinweis:

Wenn Invoke-JobProcessor.ps1 nur mit Queue gestartet wird, kann er potentiell mehrere Dateien aus queues/incoming verarbeiten.

Für die API-/Hangfire-Ansteuerung soll deshalb geprüft werden, ob ein optionaler Parameter für eine gezielte Datei bereits existiert.

Falls nicht vorhanden, soll die PowerShell-Seite minimal erweitert werden, damit ein konkretes Queue-File gezielt verarbeitet werden kann.

Empfohlenes neues Argument:

QueueFileName

oder:

SourceFileName

Ziel:

Invoke-JobProcessor.ps1 -Queue standard -QueueFileName "GUID_Action_User_pshjob_.csv" -CorrelationId "GUID" -OutputJson

Damit wird verhindert, dass ein API-Request versehentlich mehrere fremde Pending-Jobs mit derselben CorrelationId verarbeitet.

Wenn die PowerShell-Seite nicht im aktuellen Workspace enthalten ist, dokumentiere diesen Punkt als zwingende Voraussetzung und implementiere die API so, dass QueueFileName bereits gespeichert und an den Runner übergeben wird.

JobProcessorRunner erstellen

Erstelle:

IJobProcessorRunner
JobProcessorRunner

Aufgabe:

Hangfire ruft diesen Runner im Hintergrund auf.

Methode:

RunAsync(Guid jobRequestId)

Ablauf:

1. JobRequest aus IJobStatusStore laden.
2. Status ProcessorStarted speichern.
3. PowerShell-Prozess starten:

    PowerShellExecutable
    -NoProfile
    -ExecutionPolicy Bypass
    -File InvokeJobProcessorPath
    -Queue {Queue}
    -CorrelationId {JobRequestId}
    -OutputJson
    -QueueFileName {SourceFileName}

Falls QueueFileName auf PowerShell-Seite noch nicht existiert, diesen Punkt als TODO markieren und alternativ vorerst ohne QueueFileName starten. Aber dokumentiere das Risiko, dass sonst mehrere Dateien der Queue verarbeitet werden könnten.

5. stdout vollständig lesen.
6. stderr vollständig lesen.
7. ExitCode speichern.
8. stdout als AutomationJobProcessorResult parsen.
9. stderr separat als technisches Log/Event speichern.
10. finalen Status aus result.status übernehmen.
11. FinalResultJson speichern.
12. StatusEvent für finalen Status speichern.
13. Bei JSON-Parsefehler Status Failed setzen.
14. Bei Timeout Prozess beenden und Status Timeout setzen.

Wichtig:

stdout ist die primäre maschinenlesbare Quelle.

stderr darf nie als JSON interpretiert werden.

stderr darf stdout nicht vermischen.

ExitCode ist technisches Zusatzsignal, aber der fachliche Status kommt aus OutputJson.

Wichtigste Zielregel

Die bestehende ASP.NET Core MVC Applikation bleibt die Annahme- und Queue-Erzeugungsoberfläche.

Die bestehende PowerShell-JobEngine bleibt die fachliche AutomationEngine.

Hangfire verbindet beide Welten:

MVC Controller
  → IamQueueService schreibt Queue-Datei
  → AutomationEngineScheduler erstellt JobRequest
  → Hangfire startet JobProcessorRunner
  → JobProcessorRunner ruft Invoke-JobProcessor.ps1 auf
  → OutputJson wird ausgewertet
  → JobController liefert Status und Events

Keine fachliche Doppelimplementierung in C#.

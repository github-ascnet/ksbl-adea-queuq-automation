K = Ausgangslage

Der generische UrgentRecipientAttributeChange-Flow ist bereits abgeschlossen:

Invoke-UrgentRecipientAttributeChange
  → generische AD-/Recipient-Attributänderung

UrgentRecipientService
  → Allowlist
  → Set/Clear
  → AD-Gateway-Nutzung

proxyAddresses
  → bewusst nicht generisch unterstützt
  → keine implizite Replace-Semantik

Diese Logik bleibt unverändert.

Jetzt soll die vollständige dringende Hospis-Inaktivierung ergänzt werden.

Die Legacy-Logik macht sinngemäss:

1. AD-Benutzer deaktivieren.
2. Falls homeMdb gesetzt ist:
   Mailbox Features / Hidden State auf HIDE setzen, sofern vorhandene Funktion existiert.
   MailboxAutoReplyConfiguration setzen.
   EMailRevocations.ValidTo in SQL auf GetDate setzen.
3. Bestimmte Gruppenmitgliedschaften entfernen:
   CN=TPL-
   CN=GG-KSBL-VDI-Remote
   CN=GG-OneSign
4. extensionAttribute6 löschen.
5. msDS-cloudExtensionAttribute15 löschen.
6. Description setzen:
   Inaktiviert (Urgent) am yyyy-MM-dd von CurrentUserName
7. Bei ActionType Inaktivieren:
   SQL Stored Procedure ausführen:
   usp_create_urgent_inaktivieren_transaction

Wichtig:

TenantState.psm1 bleibt unberührt.
Set-TenantState darf nicht produktiv verwendet werden.
Die Entra-Sync-Deaktivierung erfolgt über das Leeren von msDS-cloudExtensionAttribute15 im OnPrem AD.

A = Architekturvorgaben

Die Zielkette soll so aussehen:

Invoke-InactivateHospisPerson
  → validiert Payload
  → ruft HospisPersonService.UrgentInactivation auf
  → gibt strukturiertes JobResult zurück

HospisPersonService.UrgentInactivation
  → orchestriert die fachlichen Schritte
  → ruft ActiveDirectoryGateway
  → ruft ExchangeOnPremGateway
  → ruft SqlGateway oder HospisSqlService
  → sammelt Ergebnisse und Fehler

ActiveDirectoryGateway
  → deaktiviert User
  → löscht Attribute
  → setzt Description
  → entfernt Gruppenmitgliedschaften

ExchangeOnPremGateway
  → setzt MailboxAutoReplyConfiguration
  → optional Mailbox Hidden / Feature-State, falls bereits bestehende Funktion existiert

SqlGateway / HospisSqlService
  → schliesst EMailRevocations
  → ruft usp_create_urgent_inaktivieren_transaction auf

Nicht erlaubt:

Direkte Set-ADUser-Aufrufe im Handler.
Direkte Set-MailboxAutoReplyConfiguration-Aufrufe im Handler.
Direkte Invoke-Sqlcmd-Aufrufe im Handler.
TenantState.psm1 verändern oder verwenden.
ExchangeOnlineGateway verwenden.
Connect-ExchangeOnline verwenden.
Neue ExchangeOnline-Logik einbauen.
JobEngine.psm1 ändern.
usecases.json ändern, ausser ein bestehender offensichtlicher Fehler wird durch Tests nachgewiesen.
Invoke-UrgentRecipientAttributeChange erneut umbauen.
UrgentRecipientService erneut umbauen, ausser eine bereits vorhandene Funktion wird nur genutzt.

Wichtigste Zielregel

Der Legacy-Code wird jetzt nur für die vollständige dringende Hospis-Inaktivierung migriert.

Invoke-UrgentRecipientAttributeChange bleibt unverändert und ist nicht Teil dieses Umbaus.

Exchange Online wird in diesem Prozess nicht verwendet.
